# LayerCake Moonshot V2 development report

Status: **NOT YET PROVEN**. All figures below are development evidence from one RTX 3080
Laptop GPU and an Intel Family 6 Model 154 CPU. The independent certificate is authoritative.

## What was built

- Exact `prefill`, `decode_step`, `decode_many`, reset, and safe state serialization for
  both foundation generations, plus an exact KV-cached transformer path.
- A 3.513M-parameter multiscale V2 foundation with 559,584 active parameters (15.93%).
- A fixed semantic canonical ABI and a 44,705-parameter host-conditioned fusion cake.
- Three actual hosts: seeds 9201 and 9202 at 3.513M parameters, and seed 9203 at 2.688M.
- A signed, safetensors-only `.cake` installation path with exact archive preservation.
- A learned local semantic router, CPU stateful runtime, CUDA cached baseline, and
  TorchScript export.
- Fourteen locked architecture ablations across three seeds, with failures retained.

## Data and training

The English data is WikiText-103 raw, normalized with NFKC, exact-record deduplicated,
language-filtered, and split before training. The development train set is exactly
10,000,000 bytes; a 100,000,000-byte medium corpus was prepared but not trained. The
one-billion-byte proof tier was not available. Sampled 64-byte cross-split overlap was zero.

The Python corpus assigns complete installed open-source distributions to disjoint splits:
PyTorch, NumPy, Transformers, and Datasets for training; huggingface-hub for validation;
and scikit-learn for test. Licenses are Apache-2.0 or BSD-3-Clause. This is a legally usable
development corpus, not a claim of CodeSearchNet benchmark coverage.

LayerCake's raw foundation wall time was 5.61x lower, but it did not reach matched quality;
therefore the training-efficiency gate fails. Cake training was only 3.92x faster than
full transformer adaptation, below the locked 5x target, and task quality was worse.
All five 128-byte greedy English samples also triggered the four-gram repetition check,
so the certificate does not treat the development core as a useful general generator.

## Quality and portability

The Python cake changed held-out code BPB from 7.1331 to 4.4568, beat random and shuffled
weights, and depended measurably on both canonical components. It nevertheless generated
zero parseable completions on eight ordinary syntax prompts, so its five-fold error gate
fails. It also changed unrelated English BPB from 2.6966 to 2.9013, outside the locked
two-percent retention tolerance.

Post-training dynamic qint8 quantization preserved the BPB result (4.4566 versus 4.4568
fp32). This is a PyTorch packed-runtime check, not a native int8 kernel or a quantized
portable package claim.

The exact same signed archive was installed, removed, reinstalled, and executed on all
three host checkpoints. Every one of the source host's 32 predeclared sequence-level BPB
successes was retained on both receivers without receiver data or calibration. This is
real BPB-level functional portability and exact payload/install losslessness. It is not
the required task-capability portability because there were no source task successes to
retain; the final gate is therefore `INVALID_EVIDENCE`.

## Runtime evidence

One-thread incremental decode sustained a 2,178 bytes/s median over 1,024 generated bytes.
Decode work was independent of prompt length; prefill scaled from about 28 ms at 64 bytes
to 1.78 s at 4,096 bytes. At a 1,024-byte prompt, exact incremental decoding was 72.36x
faster than repeated full-context decoding over 64 generated bytes.

The frozen direct benchmark contains 100 prompts, 256/384 generated-byte buckets, greedy
and sampled modes, three repetitions, randomized order, learned routing, and 600 raw rows.
CPU LayerCake measured 1,930 useful bytes/s median versus 280 on the fp16 cached GPU
transformer, a 6.88x raw throughput ratio; median end-to-end latency was 215.7 ms versus
1,277.5 ms. Routing was 0.88% of LayerCake latency. The result is deliberately
`INVALID_EVIDENCE`: it has one real cake rather than three domains, and the ordinary
Python task threshold failed. No quality-matched CPU-over-GPU claim is made.

The CPU path currently uses PyTorch/oneDNN with a persistent batch-one state. No custom
C++/Rust extension, int4 kernel, energy meter, temperature sensor, or thread-affinity
study was completed. The GPU baseline uses pre-normalization, SwiGLU, tied embeddings,
SDPA, fp16, and a per-layer KV cache. The mobile artifact reloads exactly under
TorchScript, but no physical mobile execution was available.

## Failure analysis and next experiment

The limiting issue is quality, not incremental decoding or package transport. The V2
foundation's local/recurrent state learns quickly but remains 0.427 BPB behind the
same-scale BPE transformer; the low-quality host then forces a very small cake to learn
too much syntax through a bounded residual. Search also failed to promote a candidate
under its locked active-fraction/quality criteria.

The single next experiment is to keep `lc-canonical-byte-semantics/2` frozen, train the
selected foundation on the prepared 100M-byte tier with three seeds and a matched
transformer, and stop unless English BPB becomes non-inferior. Only then should the Python
cake be retrained and required to achieve nonzero hidden unit-test and syntax success
before adding other domains or rerunning the headline CPU-versus-GPU gate.
